Claude Code를 사용해 개발하다 보면 반복적인 git 작업이 눈에 띕니다.

- 변경사항을 보고 적절한 커밋 메시지를 고민하는 일
- PR 본문을 커밋 히스토리 보면서 손으로 쓰는 일
- 코드 리뷰 코멘트를 하나씩 읽고 반영한 뒤 답글 다는 일
- 머지 후 브랜치 정리하는 일

이 작업들을 Claude Code의 **Custom Skill**로 묶어두면, `/git-commit` 같은 슬래시 커맨드 하나로 처리할 수 있습니다. 이 글에서는 TierCup 프로젝트에서 직접 만들어 쓰고 있는 6개의 git 스킬을 소개합니다.

## Custom Skill이란?

Claude Code는 `~/.claude/skills/<skill-name>/SKILL.md` 파일에 Markdown으로 지시사항을 작성해두면, `/skill-name`으로 호출할 수 있는 커스텀 슬래시 커맨드를 만들 수 있습니다.

스킬은 단순 텍스트 명령이 아니라 **Claude가 실행하는 절차서**입니다. 각 단계마다 어떤 명령어를 실행하고, 어떤 조건에서 멈추며, 결과를 어떻게 가공할지를 기술합니다.

```
~/.claude/skills/
  git-commit/
    SKILL.md
  git-pr/
    SKILL.md
  git-code-review-pr/
    SKILL.md
  git-code-review-apply/
    SKILL.md
  git-deep-review-pr/
    SKILL.md
  git-merge/
    SKILL.md
```

## 스킬 1: `/git-commit` — 변경사항을 논리 단위로 커밋

#### 문제

`git add -A && git commit -m "update"`로 몰아서 커밋하거나, 반대로 커밋 단위를 어떻게 나눌지 고민하다 시간을 씁니다.

#### 동작 방식

`/git-commit`을 실행하면 Claude가 다음 절차를 수행합니다.

1. `git status` + `git diff HEAD`로 전체 변경사항 파악
2. 변경의 성격(feat, fix, chore, docs 등)과 관심사에 따라 커밋 단위 분류
3. 단위별로 커밋 메시지 초안 작성 (50자 이내 subject, 복잡한 변경엔 body 추가)
4. 파일별로 `git add` + `git commit` 실행
5. `git log --oneline`으로 결과 요약

서로 다른 타입이 섞이거나(feat + fix), 논리적으로 독립된 파일 그룹이 있으면 분할 커밋을 합니다.

```
예시 분류:
- package.json, package-lock.json → chore: dnd-kit 패키지 설치
- components/TierBoard.tsx        → feat: TierBoard 컴포넌트 추가
- app/create/page.tsx             → feat: create 페이지에 TierBoard 연결
```

#### 커밋 메시지 형식

```
<type>: <subject>          # 50자 이내, 현재형

- <bullet 1>               # 변경 파일 3개 이상이면 body 추가
- <bullet 2>

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

#### 핵심 구조

```markdown
## 2단계: 변경사항 분석

diff를 읽고 다음을 파악합니다:
- 어떤 파일들이 변경되었는가 (신규/수정/삭제)
- 각 파일에서 어떤 내용이 추가/수정/삭제되었는가
- 변경의 성격 (기능 추가, 버그 수정, 설정 변경 등)

## 3단계: 논리 단위 분류

아래 기준으로 커밋 단위를 결정합니다:
- 서로 다른 type이 섞여 있음 (예: feat + fix, chore + feat)
- 논리적으로 독립적인 목적을 가진 파일 그룹
- 분리하면 각 커밋이 더 명확하고 revert가 쉬워지는 경우
```

## 스킬 2: `/git-pr` — 커밋 히스토리로 PR 자동 생성

#### 문제

PR 본문을 매번 손으로 쓰는 게 번거롭고, 커밋 목록이나 테스트 계획을 빠뜨리기 쉽습니다.

#### 동작 방식

`/git-pr`을 실행하면 Claude가 다음 절차를 수행합니다.

1. 현재 브랜치가 `main`/`develop`이면 중단
2. `git log develop..HEAD --oneline`으로 커밋 목록 수집
3. 브랜치명에서 PR 제목 파생 (`feature/task-001-dev-environment-setup` → `chore: Task 001 — 개발 환경 설정`)
4. 관련 GitHub 이슈 번호 자동 탐색 (`gh issue list --search`)
5. Summary / Commits / Test plan / Closes #N 포함한 PR 본문 작성
6. 브랜치 prefix에 맞는 라벨 적용 (없으면 생성)
7. `gh pr create`로 PR 생성, 프로젝트 보드에 연결

#### PR 본문 구조

```markdown
## Summary

- <변경사항 핵심 1>
- <변경사항 핵심 2>

## Commits

- `abc1234` feat: TierBoard 컴포넌트 추가
- `def5678` test: TierBoard 단위 테스트 추가

## Test plan

- [ ] 티어보드에서 항목 드래그 앤 드롭이 동작한다
- [ ] 빈 티어에 항목을 추가할 수 있다

Closes #42

🤖 Generated with Claude Code
```

#### 핵심 구조

```markdown
## 4단계: PR 본문 작성

#### 이슈 번호 조회

브랜치명에서 Task 번호를 추출해 연결할 이슈 번호를 찾습니다:

TASK_NUM=$(git branch --show-current | grep -oE 'task-[0-9]+' | grep -oE '[0-9]+')

if [ -n "$TASK_NUM" ]; then
  ISSUE_NUM=$(gh issue list --search "Task $(printf '%03d' $((10#$TASK_NUM)))" \
    --json number,title --jq '.[0].number // empty')
fi
```

## 스킬 3: `/git-code-review-pr` — 리뷰를 PR 인라인 코멘트로 게시

#### 문제

Claude에게 코드 리뷰를 받아도 결과가 채팅창에 텍스트로만 남아서, PR에 인라인 코멘트로 남기려면 수동으로 복사해야 합니다.

#### 동작 방식

`/git-code-review-pr 42`를 실행하면 Claude가 다음 절차를 수행합니다.

1. PR의 변경 파일 목록과 patch 수집
2. `code-reviewer` 서브에이전트를 실행해 리뷰 생성
3. 각 지적사항의 파일 위치를 diff position으로 변환
4. `gh api`로 인라인 코멘트 게시

GitHub PR 인라인 코멘트는 파일 내 절대 줄 번호가 아닌, **diff 내 상대 position**을 사용합니다.

```
patch 예시:
@@ -10,6 +10,8 @@         ← 이 줄은 position 카운트 제외
 context line              ← position 1
-deleted line              ← position 2 (삭제된 줄엔 코멘트 불가)
+added line                ← position 3
 context line              ← position 4
```

position이 불확실한 항목은 인라인 코멘트를 건너뛰고 PR 일반 코멘트로 보완합니다.

#### 심각도 기준

| 이모지 | 의미 |
|--------|------|
| 🔴 | 높음 — 즉시 수정 필요 |
| 🟡 | 중간 — 개선 권장 |
| 🟢 | 낮음 — 선택적 개선 |

## 스킬 4: `/git-code-review-apply` — 리뷰 코멘트를 코드에 자동 반영

#### 문제

PR 리뷰 코멘트를 하나씩 읽고, 수정하고, 답글 달고, 커밋하는 과정이 반복됩니다.

#### 동작 방식

`/git-code-review-apply 42`를 실행하면 Claude가 다음 절차를 수행합니다.

1. 인라인 코멘트 전체 수집 — `in_reply_to_id`가 있는 reply는 제외하고 원본만
2. PR 일반 코멘트도 확인 (position 불명확으로 인라인이 아닌 코멘트)
3. 각 코멘트의 수정 사항을 코드에 반영
4. `tsc → lint → test → build` 전체 검증
5. 통과 후 자동 커밋 + 푸시
6. 적용된 코멘트에 커밋 링크 포함한 답글 게시

이미 답글이 달린 원본 코멘트는 재처리하지 않아 스킬을 여러 번 실행해도 중복 반영되지 않습니다.

#### 핵심 구조

```markdown
## 1단계: PR 코멘트 fetch

# in_reply_to_id가 null인 것만 원본 코멘트
gh api repos/$REPO/pulls/<PR번호>/comments \
  --jq '[.[] | select(.in_reply_to_id == null)] | .[] | {id, path, line: .original_line, body}'

# 이미 처리된 코멘트 ID 목록 (reply가 달린 것)
gh api repos/$REPO/pulls/<PR번호>/comments \
  --jq '[.[] | select(.in_reply_to_id != null) | .in_reply_to_id] | unique'
```

## 스킬 5: `/git-deep-review-pr` — 두 리뷰어 교차 리뷰

#### 문제

단일 모델의 리뷰는 놓치는 부분이 있습니다. 특히 아키텍처 수준의 문제와 세부 코드 품질 문제를 동시에 잡기 어렵습니다.

#### 동작 방식

`/git-deep-review-pr 42`를 실행하면 두 리뷰어가 순차적으로 동작합니다.

```
1단계: PR 정보 수집
   ↓
2단계: Codex 리뷰 (/codex:review --base <baseRefName>)
   ↓
3단계: code-reviewer 에이전트가 Codex 결과를 재검토
       (동의 / 반박 / 보완 의견 추가)
   ↓
4단계: 최종 이슈 목록으로 인라인 코멘트 게시
```

Codex가 놓친 문제를 code-reviewer가 보완하고, Codex가 과도하게 지적한 부분을 code-reviewer가 반박하는 교차 검토 구조입니다.

#### 리뷰 코멘트 형식

```
🔴 **타입 안전성 문제**

any 타입 사용으로 런타임 오류 발생 가능

**Codex**: `items: any[]`를 구체적인 타입으로 교체 권장
**검토**: 동의. `Item[]` 타입으로 교체 시 하위 컴포넌트 타입도 함께 수정 필요
```

## 스킬 6: `/git-merge` — Squash merge + 브랜치 정리

#### 문제

PR 머지 후 원격/로컬 브랜치 삭제, base 브랜치 pull을 매번 수동으로 합니다.

#### 동작 방식

`/git-merge`를 실행하면 Claude가 다음 절차를 수행합니다.

1. 현재 브랜치와 PR 상태 확인 (충돌 여부, 머지 가능 여부)
2. `CONFLICTING`이면 즉시 중단, `UNKNOWN`이면 10초 후 재확인
3. 머지 정보 출력 후 확인
4. `gh pr merge --squash --delete-branch` 실행 (원격 브랜치 자동 삭제)
5. base 브랜치로 체크아웃 + 로컬 feature 브랜치 삭제
6. `git pull origin <base>`로 최신화

Squash merge를 사용하는 이유는 feature 브랜치의 WIP 커밋들이 main/develop에 그대로 들어가지 않도록 하기 위함입니다.

## 전체 워크플로우

```
구현 완료
   ↓
/git-commit              # 변경사항 → 논리 단위 커밋
   ↓
git push
   ↓
/git-pr                  # 커밋 히스토리 → PR 자동 생성
   ↓
/git-deep-review-pr <N>  # Codex + code-reviewer 교차 리뷰
   ↓
/git-code-review-apply <N>  # 리뷰 반영 + 자동 커밋 + 답글
   ↓
/git-merge               # Squash merge + 브랜치 정리
```

## 스킬 파일 작성 팁

#### 1. context: fork 설정

```yaml
---
name: git-commit
context: fork
---
```

`context: fork`를 설정하면 스킬이 포크된 Claude 인스턴스에서 실행됩니다. 스킬 실행 중 발생하는 대량의 git 출력이 메인 대화 컨텍스트를 오염시키지 않습니다.

#### 2. 조기 종료 조건 명시

스킬 초반에 `main`/`develop` 브랜치 체크, PR 존재 여부, 인자 유무 등을 확인하고 조건이 맞지 않으면 즉시 중단하도록 작성합니다. Claude가 잘못된 상태에서 작업을 이어가는 것을 방지합니다.

#### 3. 검증 단계 포함

코드를 변경하는 스킬(`git-code-review-apply`)에는 반드시 검증 단계를 포함시킵니다.

```bash
npx tsc --noEmit && npm run lint && npm test && npm run build
```

## 마치며

6개 스킬이 각각 독립적으로도 쓰이지만, 묶이면 구현 완료부터 머지까지 Claude가 흐름을 이어받습니다. 커밋 분류, PR 생성, 교차 리뷰, 리뷰 반영, 브랜치 정리 — 판단이 필요한 부분만 사람이 개입하고, 나머지는 스킬이 처리합니다.

스킬 파일은 프로젝트나 팀에 맞게 커스터마이즈할 수 있어서, 커밋 컨벤션이나 PR 템플릿이 다르다면 SKILL.md를 그에 맞게 수정하면 됩니다.

스킬 전체 코드는 [GitHub](https://github.com/BurningFalls/tiercup/tree/develop/.claude/skills)에 공개되어 있습니다.
