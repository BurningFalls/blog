Claude Code를 사용해 TierCup 프로젝트를 개발하던 중 이상한 점을 발견했습니다.

단순한 git 명령 4~5개만 실행하는 스킬이, 수백 번 호출되는 코드 리뷰 스킬보다 **총 토큰 소모가 더 많았습니다.**

## 7일간의 API 사용 현황

Claude Code에는 `session-report` 스킬이 있습니다. `~/.claude/projects` 아래 저장된 대화 이력을 분석해서 세션별 토큰 소모, 스킬별 호출 비용, 캐시 히트율 등을 시각화해주는 도구입니다.

2026-05-08 ~ 2026-05-14, 7일간의 TierCup 개발 세션을 분석했습니다.

```
총 토큰:     169.8M
세션 수:     56회
API 호출:   3,308회
캐시 히트율: 95.3%
```

스킬별 토큰 소모를 정리하면 이렇습니다.

| 스킬 | 호출 수 | 총 토큰 | 회당 평균 |
|------|---------|---------|-----------|
| git-merge | 16회 | 5.72M | **357k** |
| code-review-pr | 234회 | 3.83M | 16k |
| playwright-cli | 2회 | 4.35M | 2,178k |
| git-pr | 4회 | 1.11M | 278k |
| git-commit | 7회 | 1.11M | 158k |

`git-merge`가 16회 호출로 `code-review-pr` 234회보다 **총 토큰이 많습니다.** 직관적으로 이해가 안 됩니다. `git-merge`는 브랜치 확인 → squash merge → 브랜치 삭제 → pull, 총 4~5개의 bash 명령만 실행하는 단순한 스킬입니다.

서브에이전트도 살펴봤습니다.

| 서브에이전트 | 호출 수 | 총 토큰 | 회당 평균 |
|-------------|---------|---------|-----------|
| general-purpose | 324회 | 26.1M | 80k |
| Explore | 30회 | 17.2M | **572k** |
| Plan | 6회 | 2.4M | 403k |

**Explore 서브에이전트**가 회당 572k로 단일 호출 비용이 가장 높습니다. 코드베이스 탐색 시 파일 검색 결과를 대량으로 읽어오는 구조 때문입니다.

## 원인: 컨텍스트가 비용을 결정한다

Claude는 stateless합니다. 이전 대화를 "기억"하지 않고, **매 API 호출마다 현재 대화 전체를 처음부터 다시 전송**합니다. 즉, 스킬 자체의 복잡도가 아니라 **호출 시점의 대화 누적량**이 토큰 비용을 결정하는 구조입니다.

```
[작업 시작] plan task: Task 009 구현  ← 컨텍스트 5만 토큰
[중반]      파일 수정, 리뷰, 디버깅   ← 누적 15만 토큰
[작업 완료] /git-merge               ← 15만 토큰이 그대로 딸려서 전송됨
```

`git-merge`는 항상 작업 세션 **후반부**에 호출됩니다. 브랜치 전략 논의, 파일 수정, 코드 리뷰까지 끝난 뒤에 실행하기 때문입니다. 앞선 대화 전체가 컨텍스트로 딸려가서 API 요청 1회당 수십만 토큰이 전송됩니다.

반면 `code-review-pr`은 보통 `/clear` 직후 새 세션에서 실행됩니다. 리뷰만 하는 세션이라 컨텍스트가 짧습니다. 234회 호출해도 총 토큰이 3.83M에 그치는 이유입니다.

스킬의 로직이 아니라 **호출 타이밍**이 비용을 결정하고 있었습니다.

실제로 session-report의 `cache_breaks` 데이터를 보면, 비용이 특히 높았던 호출이 모두 긴 작업 세션 도중임을 확인할 수 있습니다.

```
# 2026-05-12 — cache break 발생 시점
/git-commit  →  커밋 완료 보고
/git-merge   ← 이 시점에 134k 토큰 uncached 전송 (총 컨텍스트 117k↑)
"자동으로 처리하지 않은 경우에만 실행으로 바꿔줘..."
```

긴 작업 직후 `/git-merge`를 호출했고, 그 시점에 쌓인 대화 전체가 한 번에 전송되면서 cache break(새로 캐시 생성)가 발생했습니다.

캐시 히트율 95.3%는 좋은 수치지만, 캐시는 **재계산 비용**을 줄일 뿐 전송되는 토큰 수 자체는 동일합니다.

```
총 입력 토큰: 168.9M
  - uncached:     30,961    (0.02%)
  - cache_create: 7,969,303  (4.71%)  ← 새로 캐시 생성
  - cache_read:  160,910,799 (95.26%) ← 캐시에서 읽음
```

cache_read 토큰은 재계산 없이 읽혀오지만, 여전히 **160.9M 토큰이 전송됩니다.** 비용을 줄이려면 캐시보다 컨텍스트 크기 자체를 줄이는 것이 더 직접적입니다.

## 해결: /clear 하고 호출하라

**1. git-merge 전 `/clear` 실행**

```
/clear
/git-merge
```

작업이 끝난 뒤 컨텍스트를 초기화하고 머지합니다. git-merge에는 이전 대화 맥락이 필요 없습니다.

**2. Explore 대신 직접 검색**

파일 경로나 심볼 위치가 이미 알려진 경우, Explore 서브에이전트 없이 `grep` / `Read`를 직접 사용합니다. 회당 572k 비용을 피할 수 있습니다.

**3. 단순 작업은 새 세션에서 실행**

머지, PR 생성, 커밋처럼 이전 대화 맥락이 필요 없는 작업은 `/clear` 후 실행하는 습관이 유효합니다.

## 실제 적용 결과

`/clear → /git-merge` 패턴을 2026-05-15부터 적용하고, 2026-06-01에 다시 측정했습니다. (측정 기간: 2026-05-14 ~ 2026-06-01, 18일)

| 지표 | Before (5/8~5/14, 7일) | After (5/14~6/1, 18일) |
|------|------------------------|------------------------|
| git-merge 호출 수 | **16회** | **13회** |
| git-merge 총 토큰 | 5,715,374 | 2,102,140 |
| **회당 평균 토큰** | **357k** | **162k** |
| **변화** | — | **-54.6%** |

호출 횟수와 측정 기간이 다르므로 총 토큰 비교는 의미가 없습니다. 최적화 효과를 순수하게 반영하는 지표는 **회당 평균 토큰**입니다. "한 번 호출할 때 얼마나 작은 컨텍스트로 실행됐는가"가 줄어야 하기 때문입니다.

회당 평균이 357k → 162k로 **54.6% 감소**했습니다. `/clear` 한 줄 추가만으로 git-merge 1회당 195k 토큰을 절감한 셈입니다.

After 기간의 서브에이전트 현황도 참고로 보면:

| 서브에이전트 | 호출 수 | 회당 평균 |
|-------------|---------|-----------|
| Explore | 27회 | **759k** (Before 572k) |
| general-purpose | 134회 | 237k (Before 80k) |
| Plan | 2회 | 345k |

Explore는 오히려 회당 비용이 올랐습니다. 이 기간에는 `/clear` 패턴을 git-merge에만 적용했고, Explore 최적화(직접 grep 사용)는 아직 적용 전이었습니다. 다음 최적화 대상입니다.

## 일반화

일반적인 소프트웨어에서는 "단순한 작업 = 낮은 비용"이 성립합니다. LLM 기반 도구에서는 그렇지 않습니다. **단순한 명령도 대화 누적량이 많으면 비용이 높아집니다.**

이 원칙은 Claude Code 외에도 적용됩니다.

- **LLM API를 직접 호출하는 서비스**: 대화 이력을 그대로 누적하면 긴 세션에서 비용이 선형이 아니라 제곱에 가깝게 증가합니다. 대화 요약, 슬라이딩 윈도우, 관련 메시지만 선택 전송 등의 트리밍 전략이 필요합니다.
- **Multi-turn 챗봇**: 컨텍스트 윈도우 한계가 오기 전에 비용 폭발이 먼저 옵니다. 어느 시점에서 컨텍스트를 압축할지를 설계 단계에서 결정해야 합니다.
- **AI 코딩 도구 장기 세션**: 한 세션에서 여러 작업을 이어서 하면, 후반 작업의 비용이 초반보다 몇 배 높아집니다. 작업 단위로 `/clear`를 끊는 것이 비용과 품질 모두에 유리합니다.

비용 최적화의 출발점은 기능 단위가 아니라 **호출 시점의 컨텍스트 크기**를 보는 것입니다.
